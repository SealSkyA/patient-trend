# Requirements Document

## Introduction

患者检查报告趋势分析系统（Patient Report Trend Analyzer）是一个面向个人和家庭的数字健康管理平台。系统将分散、非结构化的检查报告（纸质/PDF/图片）转化为结构化、可对比的时序数据，自动生成指标趋势曲线图，提供早期预警与决策支持，建立患者终身可追溯的电子检查档案。

## Glossary

- **患者报告趋势分析系统**: 本系统，用于管理患者检查报告、生成趋势图表、提供健康洞察
- **指标**: 检查项目中的具体测量值，如血糖、血脂、尿酸等
- **归一化**: 将同一指标的不同名称别名统一为标准名称，如"空腹血糖""FBG"统一为"血糖-空腹"
- **异常值**: 指标数值超出参考范围的情况
- **趋势线**: 基于历史数据绘制的线性回归辅助线，显示指标变化方向
- **EARS**: Easy Approach to Requirements Syntax，需求编写标准语法
- **OCR**: 光学字符识别，用于从图片/PDF中提取文本
- **HIS**: 医院信息系统 (Hospital Information System)
- **LIS**: 实验室信息系统 (Laboratory Information System)
- **HL7/FHIR**: 医疗健康数据交换标准

## Requirements

### Requirement 1: 用户注册与登录

**User Story:** AS 患者或家属, I want 注册账号并安全登录系统, so that 我可以管理个人和家人的检查报告档案。

#### Acceptance Criteria

1. WHEN 用户提供有效手机号和验证码, 系统 SHALL 完成用户注册并创建账号
2. WHEN 用户输入正确的账号凭证, 系统 SHALL 验证身份并授予访问权限
3. IF 用户连续5次输入错误密码, 系统 SHALL 锁定账号15分钟后允许重试
4. WHILE 用户处于已认证状态, 系统 SHALL 保持会话有效并允许访问授权功能
5. WHERE 用户忘记密码, 系统 SHALL 提供通过手机号验证码重置密码的流程

### Requirement 2: 多患者档案管理

**User Story:** AS 账号持有人, I want 创建和管理多个患者档案, so that 我可以为家人（如父母、子女）分别维护独立的检查记录。

#### Acceptance Criteria

1. WHEN 已登录用户点击"添加患者", 系统 SHALL 要求输入患者姓名、性别、出生日期并创建独立档案
2. WHEN 用户选择特定患者档案, 系统 SHALL 切换上下文并仅显示该患者的检查数据
3. IF 用户尝试删除患者档案且该档案包含检查记录, 系统 SHALL 要求二次确认并提示数据将一并删除
4. WHILE 用户处于患者档案视图, 系统 SHALL 在页面顶部显示当前选中的患者名称

### Requirement 3: 手动录入检查指标

**User Story:** AS 患者或家属, I want 手动输入检查指标数值, so that 我可以在没有电子版报告时快速记录关键指标。

#### Acceptance Criteria

1. WHEN 用户选择患者并点击"手动录入", 系统 SHALL 展示指标选择表单，包含指标名称、数值、单位、检查日期字段
2. WHEN 用户输入指标名称的前3个字符, 系统 SHALL 自动匹配指标库并显示候选列表供选择
3. IF 用户输入的数值超出医学合理范围（如血糖>50 mmol/L）, 系统 SHALL 弹出二次确认对话框提示用户核实
4. WHERE 用户提交手动录入, 系统 SHALL 保存记录并更新该患者的指标时间序列数据
5. WHEN 用户需要添加检查备注, 系统 SHALL 提供可选的备注文本框（如"检查前未空腹"）

### Requirement 4: 图片/PDF上传与OCR识别（P2 延后实现）

**User Story:** AS 患者或家属, I want 上传检查报告图片或PDF, so that 系统可以自动提取检查项目数据，减少手动输入工作量。

> **注**：本需求标记为 P2 优先级，首期实现以人工录入为主，OCR功能后续迭代加入。

#### Acceptance Criteria

1. WHEN 用户点击"上传报告", 系统 SHALL 接受JPG、PNG、PDF格式文件，单文件大小限制50MB以内（P2实现）
2. WHEN 文件上传完成, 系统 SHALL 调用OCR服务并返回识别预览结果（P2实现）
3. IF OCR识别置信度低于70%, 系统 SHALL 在预览界面标记低置信度区域并提示用户人工核对（P2实现）
4. WHILE 用户处于识别预览页面, 系统 SHALL 允许用户修改任意识别字段（P2实现）
5. WHERE 用户确认识别结果, 系统 SHALL 保存结构化数据并关联到指定患者档案（P2实现）

### Requirement 4-Alt: 智能快捷录入（P0 核心功能，替代OCR）

**User Story:** AS 患者或家属, I want 快速便捷地录入检查指标, so that 我可以在无OCR支持时高效完成数据输入。

#### Acceptance Criteria

1. WHEN 用户进入手动录入页面, 系统 SHALL 提供最近使用指标的历史记录快捷选择区
2. WHEN 用户开始输入指标名称的前2个字符, 系统 SHALL 实时匹配指标库并显示候选列表，支持键盘方向键和回车选择
3. IF 用户连续录入同一报告的多个指标, 系统 SHALL 自动复用已填写的检查日期和检查机构，减少重复输入
4. WHERE 用户完成单个指标录入, 系统 SHALL 提供"继续添加同级指标"和"完成并保存"两个选项，支持连续批量录入
5. WHEN 用户需要录入常规检查套餐（如肝功能全套）, 系统 SHALL 提供预置模板一键填充所有子指标空值，用户仅需填数值
6. WHILE 移动端录入, 系统 SHALL 提供数字键盘优先的输入框，避免用户在字母键盘和数字键盘间切换

### Requirement 5: 指标库管理与归一化

**User Story:** AS 系统管理员或用户, I want 维护标准指标库和管理自定义录入模板, so that 不同医院的检查项目名称可以统一归一化，且用户可以使用预设模板快速录入。

#### Acceptance Criteria

1. WHEN 系统启动, 系统 SHALL 加载预设指标库，包含血常规、尿常规、肝功能、肾功能、血脂四项、甲状腺功能等常见检查项
2. WHEN 指标库接收新指标名称, 系统 SHALL 根据别名映射表将其归一化为标准名称（如"空腹血糖""FBG"映射为"血糖-空腹"）
3. IF 用户尝试添加指标库中不存在的项目, 系统 SHALL 允许创建自定义指标并记录为用户自定义类型
4. WHERE 同一指标存在不同单位（如mg/dL和mmol/L）, 系统 SHALL 根据内置换算公式自动转换为标准单位
5. WHILE 用户查看指标详情, 系统 SHALL 显示该指标的标准参考范围，区分年龄和性别差异
6. WHEN 用户进入模板管理页面, 系统 SHALL 提供创建、编辑、删除自定义录入模板的功能
7. WHERE 用户创建自定义模板, 系统 SHALL 允许用户从指标库中选择多个指标组成模板，并保存模板名称（如"我的肝功能全套"）
8. IF 用户选择已有模板进行录入, 系统 SHALL 一键填充模板中所有指标的空值，用户仅需逐项填写数值

### Requirement 5-Alt: 预置指标模板

**User Story:** AS 用户, I want 使用系统预置的常见检查套餐模板, so that 我可以一键开始录入常规检查项目。

#### Acceptance Criteria

1. WHEN 用户进入录入页面, 系统 SHALL 展示预置模板快捷入口，包含"血常规全套"、"肝功能全套"、"肾功能全套"、"血脂四项"、"甲状腺功能"等常用模板
2. WHEN 用户点击预置模板, 系统 SHALL 自动创建一条新报告并填充模板中所有指标的空行，用户仅需逐项填写数值
3. WHERE 预置模板中的指标, 系统 SHALL 自动填充标准单位和参考范围，用户不可修改指标名称
4. IF 用户希望修改预置模板, 系统 SHALL 允许基于预置模板另存为自定义模板

### Requirement 6: 趋势曲线图生成

**User Story:** AS 患者或医生, I want 查看指标随时间变化的趋势曲线图, so that 我可以直观理解指标波动情况，辅助健康管理决策。

#### Acceptance Criteria

1. WHEN 用户进入趋势页面并选择特定指标, 系统 SHALL 聚合该患者的所有历史检查结果并生成折线图
2. WHEN 趋势图初始渲染, 系统 SHALL 在2秒内完成单患者千条指标数据的可视化绘制
3. IF 指标数值超出参考范围, 系统 SHALL 在图表上用红色标记该异常数据点
4. WHERE 用户鼠标悬停于数据点, 系统 SHALL 显示详细数值、检查日期、单位和参考区间的悬浮提示
5. WHEN 用户选择时间范围筛选器（近3个月、半年、1年、全部）, 系统 SHALL 重新渲染对应时间段内的趋势数据
6. WHILE 图表展示, 系统 SHALL 自动绘制线性回归趋势线并标注变化方向（上升/下降/平稳）

### Requirement 7: 多图对比功能

**User Story:** AS 患者或医生, I want 对比多个趋势图, so that 我可以分析不同患者之间或不同指标之间的关联性。

#### Acceptance Criteria

1. WHEN 用户选择"同指标多患者对比", 系统 SHALL 在同一坐标系中绘制多个患者的同一指标趋势线，每条线使用不同颜色标识
2. WHEN 用户选择"同患者不同指标对比", 系统 SHALL 展示双Y轴或多面板视图以支持不同量纲指标的对比
3. IF 用户选择的对比超过5条曲线, 系统 SHALL 提示图例可能影响可读性并建议减少对比数量

### Requirement 8: 个人仪表板

**User Story:** AS 患者, I want 在首页看到关键健康摘要, so that 我可以快速了解需要关注的异常指标和最新检查情况。

#### Acceptance Criteria

1. WHEN 用户登录系统, 系统 SHALL 展示个人仪表板，包含近期异常指标列表、最新报告摘要、需要关注的趋势
2. IF 某指标连续3次检测值呈上升趋势且超出参考范围, 系统 SHALL 在仪表板标记为"需关注"并显示警示图标
3. WHILE 用户查看仪表板, 系统 SHALL 提供最近5份报告的快捷入口和最新关键指标卡片
4. WHERE 用户点击异常指标卡片, 系统 SHALL 跳转至该指标的详细趋势图页面

### Requirement 9: 报告库管理

**User Story:** AS 患者或家属, I want 浏览和管理所有历史报告, so that 我可以查看原始报告、下载报告或删除错误录入。

#### Acceptance Criteria

1. WHEN 用户进入报告库页面, 系统 SHALL 按检查日期倒序显示所有报告，每份报告可折叠展开查看详细结果
2. WHEN 用户点击报告预览, 系统 SHALL 展示原始上传图片或PDF的查看器
3. IF 用户尝试删除报告, 系统 SHALL 要求二次确认并提示该报告关联的所有指标数据将一并删除
4. WHERE 用户点击下载报告, 系统 SHALL 生成包含报告详情和趋势图的PDF文件供下载

### Requirement 10: 趋势图导出与分享

**User Story:** AS 患者, I want 导出或分享趋势图表, so that 我可以在复诊时提供给医生参考。

#### Acceptance Criteria

1. WHEN 用户在趋势图页面点击"导出", 系统 SHALL 提供图片（PNG）和PDF两种导出格式选项
2. WHEN 用户点击"生成分享链接", 系统 SHALL 创建仅读的临时访问链接，默认有效期为7天
3. IF 用户需要设置分享有效期, 系统 SHALL 允许选择1天、7天、30天或自定义日期
4. WHERE 分享链接过期, 系统 SHALL 拒绝访问并提示链接已失效
5. WHILE 医生通过分享链接访问, 系统 SHALL 仅展示趋势图数据，不提供编辑或删除权限

### Requirement 11: 数据安全与隐私保护

**User Story:** AS 系统运营方, I want 确保患者数据的安全性和隐私合规, so that 系统符合国家健康数据相关法规要求。

#### Acceptance Criteria

1. WHEN 系统存储检查报告数据, 系统 SHALL 使用AES-256加密算法对敏感数据进行加密
2. WHERE 系统传输数据, 系统 SHALL 强制使用HTTPS协议，禁止明文HTTP传输
3. IF 用户尝试访问非本人或非授权的患者档案, 系统 SHALL 拒绝访问并返回403状态码
4. WHEN 用户分享给医生的链接被访问, 系统 SHALL 验证访问凭证，确保仅读权限且过期失效
5. WHERE 系统收集用户信息, 系统 SHALL 不收集身份证号等非必要敏感个人信息

### Requirement 12: 指标校验与纠错

**User Story:** AS 患者或家属, I want 修正录入错误的数据, so that 趋势图的准确性和可信度得到保障。

#### Acceptance Criteria

1. WHEN 用户发现已保存的指标数据有误, 系统 SHALL 提供编辑入口，允许修改数值、单位、日期字段
2. IF 用户修改已用于趋势图的数据点, 系统 SHALL 自动刷新相关图表并显示更新后的趋势
3. WHERE OCR识别结果存在歧义, 系统 SHALL 提供候选值列表供用户选择最匹配的选项
4. WHILE 用户进行数据校验, 系统 SHALL 高亮显示与参考范围不一致的数据项

### Requirement 13: 响应式设计与移动端适配

**User Story:** AS 移动端用户, I want 在手机上正常使用系统功能, so that 我可以随时随地录入和查看检查报告。

#### Acceptance Criteria

1. WHEN 用户使用手机浏览器访问系统, 系统 SHALL 自动适配屏幕宽度并提供触控友好的交互界面
2. WHERE 用户在微信内置浏览器打开链接, 系统 SHALL 保持完整功能并支持微信内正常渲染
3. IF 用户需要在移动端上传图片, 系统 SHALL 支持从相册选择或直接调用相机拍照
4. WHILE 用户查看趋势图, 系统 SHALL 支持双指缩放查看局部数据细节

### Requirement 14: 示例报告引导

**User Story:** AS 首次使用用户, I want 通过示例报告快速了解系统用法, so that 我可以降低学习成本并快速上手。

#### Acceptance Criteria

1. WHEN 新用户完成注册首次登录, 系统 SHALL 提供"查看示例报告"引导入口
2. IF 用户点击示例报告, 系统 SHALL 展示一份虚拟患者的完整报告数据和趋势图表
3. WHERE 用户浏览示例报告, 系统 SHALL 通过步骤提示引导用户了解录入、查看、分享等核心功能

### Requirement 15: 检查机构信息记录

**User Story:** AS 患者或家属, I want 记录每次检查的医疗机构, so that 我可以追溯报告来源并区分不同机构的参考范围差异。

#### Acceptance Criteria

1. WHEN 用户上传或录入报告, 系统 SHALL 要求选择或输入检查机构名称
2. WHERE 用户多次在同一家机构检查, 系统 SHALL 提供历史机构列表供快速选择
3. IF 不同机构对同一指标的参考范围存在差异, 系统 SHALL 以报告录入时记录的参考范围为准
